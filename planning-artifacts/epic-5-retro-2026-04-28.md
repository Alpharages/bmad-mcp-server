## Friction log resolution

### `story-5-7-skill-fix` (11 entries)

All 11 entries were addressed by story 5-7 via in-line patches to the
`clickup-create-story` and `clickup-dev-implement` skill step files.
Zero `won't-fix in 5-7` rows; all 11 received concrete diffs landing in
a single story 5-7 commit.

Three highest-severity entries:

- **`epic-picker-no-root-level-filter`** (MEDIUM). Story 5-7 amended
  `step-02-epic-picker.md` to filter enumerated Backlog tasks to
  root-level tasks only (`parent_task_id` null/absent). Under the
  same-list pivot introduced by story 5-4, subtasks of `86excfrge` now
  live alongside the epic in the Backlog — this filter prevents those
  subtasks from appearing as candidate epics in a future skill run.

- **`pwd-deviation-cwd-not-pilot-repo`** (MEDIUM). Story 5-7 added a
  cwd-assertion guard at instruction 0 of both skills' step-01 files,
  using a `.bmad-pilot-marker` sentinel file at the pilot-repo root.
  When the guard fires (cwd is not the pilot repo), it emits a `❌`
  error block and halts before any artifact is produced under the wrong
  cwd. The disclosed-deviation escape hatch (absolute-path `Read`
  substitution) is documented in the same instruction so the contract is
  visible to a future operator.

- **`step-05-in-review-literal-match-miss`** (MEDIUM). Story 5-7
  expanded `step-05-status-transition.md` instruction 4's literal match
  from `"in review"` only to a five-synonym match set (`in review`,
  `ready for review`, `code review`, `pending review`, `awaiting
review`). The first priority-ordered hit in `{list_statuses}` becomes
  `{transition_target}`; the warning block now enumerates all five
  synonyms so an operator reading the warning sees what the skill tried.

### `story-5-7-config-fix` (3 entries)

## What went well

### 1. Friction log quality gave story 5-7 a pre-sorted workload

The 22-entry structured log (severity, source story, surface, workaround,
proposed-fix-owner) gave story 5-7 a pre-sorted, pre-prioritised
translation workload. None of the 16 in-scope entries required
re-investigation of root cause at translation time — the owner-bucket
classification done in story 5-6 was sufficient to determine fix scope
without re-reading the execution stories' Dev Agent Records. This
contrasts sharply with the EPIC-3 retro, which had no pilot-collected
friction log and required the retrospective itself to surface latent risks
(see EPIC-3 §What didn't go well items 4 and 5). The capture-first /
translate-second protocol paid a visible dividend.

**Impact:** Zero rework at translation time. Story 5-7 could iterate
the `story-5-7-*` owner-queue rows and write diffs without re-opening
execution logs. The retro inherited a clean signal: 0 HIGH entries
unresolved, 0 `won't-fix` rows in the 16-entry scope.

### 2. Pilot scope discipline — no creep, no descoping

The pilot epic (`86excfrge`) stayed within the EPIC-5 guardrail of ≥3
stories / ≤~6 stories. Story 5-4 created exactly 3 subtasks covering the
lore-memory-mcp deliverable: schema, save-lesson, query-lessons — the
minimal backbone needed for the memory backend. No scope was added
mid-pilot, and no subtask was abandoned. The same-list pivot forced by
`cross-list-subtask-block` expanded the story's scope only in the
workaround dimension; EPIC-5 outcome bullet 3 was met within the original
story boundary.

**Impact:** Story 5-6's friction log had clean attribution — every entry
traced to a specific execution story (5-3, 5-4, or 5-5) without
ambiguity about which invocation caused it.

### 3. Post-5-7 refinements landed without regression

All 16 friction-log entries in the three `story-5-7-*` buckets received
concrete patches in story 5-7's diff — 7 skill step files, 1 config
TOML, 2 planning artifacts, and 1 smoke harness annotation — yet the test
baseline held at 234 passing / 0 failing. Story 5-7 produced the largest
diff in EPIC-5 and completed in a single dev session with zero `won't-fix`
rows.

**Why:** The step-file-per-responsibility architecture from EPIC-2 /
EPIC-3 made the 11 `story-5-7-skill-fix` patches surgical — one paragraph
per instruction, no cross-step rewrites. A skill with 7 steps each under
~100 lines can absorb workspace-specific friction without structural
redesign.

### 4. Team-facing artifact produced at story 5-8

Story 5-8 produced `docs/clickup-quickstart.md` covering all six required
sections plus §Where things live and §What comes next. The code-review
pass found four patches (three content gaps and one heading-order issue),
all applied in the same session. All 14 friction-log anchor links were
verified per story 5-8 §Completion Notes.

**Impact:** Any new team adopting the BMAD+ClickUp flow now has a single
linkable entry point rather than needing to cross-reference the friction
log, six story files, and multiple skill step files.

## What didn't go well

### 1. cwd deviation in both pilot execution stories

Both story 5-4 and story 5-5 ran with `pwd` pointing at the
bmad-mcp-server repo
(`/Volumes/Data/project/products/alpharages/bmad-mcp-server`) rather than
the pilot repo (`/Volumes/Data/project/products/alpharages/lore`). The
absolute-path `Read` escape hatch was invoked in both stories and
disclosed in their Dev Agent Records per AC (short-IDs:
`pwd-deviation-cwd-not-pilot-repo` and, retrospectively,
`multi-repo-cwd-handling-undocumented`).

Story 5-7 added the `.bmad-pilot-marker` cwd-assertion to both skills'
step-01 files. The post-5-7 guard would have caught the 5-4 / 5-5
deviation — the guard emits a `❌` error block at step-01 instruction 0,
halting before any artifact is produced under the wrong cwd. The
absolute-path `Read` escape hatch remains available but is now an explicit
disclosed deviation rather than a silent fall-through.

The underlying cause (Claude Code CLI's `pwd` not auto-following the
user's `cd` in a multi-repo session) remains a platform behaviour the
operator must work around. Story 5-8's §Multi-repo Claude Code sessions
documents the workaround, but the guard has not yet been exercised against
a live invocation.

### 2. DS trigger-dispatch not confirmed in Claude Code CLI mode

Friction-log entry `ds-trigger-not-dispatched-via-toml` is classified
`documented-not-fixed` in story 5-7: the post-5-7
`_bmad/custom/bmad-agent-dev.toml` top-of-file comment now explicitly
states that Claude Code CLI mode does not dispatch via the
`[[agent.menu]]` trigger entries; the agent walks the seven
`clickup-dev-implement` skill steps directly. The same pattern was
observed for the CS trigger in story 5-4.

The workaround (direct step-walk) produces the same contracted artifacts
(M1 / M2 comments, status transition, PR link), so this is not a
functional blocker. But whether the `CS` / `DS` TOML trigger model works
in Claude Code CLI at all remains unestablished. Until confirmed, the
trigger codes are Cursor / VS Code-only in practice — Claude Code CLI
users must invoke the skill steps directly by reading the skill files.

### 3. `cross-list-subtask-block` requires human workspace-admin action

HIGH-severity entry `cross-list-subtask-block` is `human-only` — the
workspace admin must enable the "Tasks in Multiple Lists" ClickApp toggle
if the cross-list layout (stories in sprint list, epic in Backlog) is
desired. The pilot pivoted to the same-list layout at execution time (AC
#3 amendment in story 5-4), and story 5-7 amended the PRD to document
both layouts. But the cross-list layout remains unavailable to any team
whose workspace has the toggle OFF, and the skills cannot detect the
toggle state before `createTask` fails with `400 ITEM_137`.

The quickstart's §Common pitfalls documents the pre-empt (`confirm toggle
state before designing layout`) and recovery (switch to same-list OR
enable the toggle). The detection gap remains — no skill guard can prevent
the `400` if the operator does not check the toggle state first.

## Key insights

1. **The "capture first, translate second" friction-log protocol worked.**
   Story 5-6's descriptive-only log with owner-bucket classification
   upstream-filtered story 5-7's scope to 16 entries without the
   translator needing to re-evaluate priority. The `human-only` bucket
   correctly identified two entries (`cross-list-subtask-block`,
   `gh-auth-wrong-account`) that would not benefit from skill edits,
   saving story 5-7 from attempting platform-level fixes beyond the
   server's control. Recommend preserving this protocol for any future
   pilot run — the owner-bucket field in the friction-log template is
   load-bearing, not ceremonial.

2. **Skill prompt content is durable under contact with a real ClickUp
   workspace.** The 22 friction entries are all either cosmetic,
   config-driven, or workspace-edge cases — none revealed a fundamental
   flaw in the skill-step architecture. The step-file-per-responsibility
   approach from EPIC-2 / EPIC-3 made all 11 `story-5-7-skill-fix`
   patches surgical (one paragraph per instruction, no cross-step
   rewrites). A skill with 7 steps each under ~100 lines can absorb
   workspace-specific friction without structural redesign.

3. **A second pilot run is recommended but is not a hard go-condition.**
   Story 5-8 deferred this decision to this retro. The case for
   requiring one: the post-5-7 `.bmad-pilot-marker` cwd-assertion guard,
   the broadened `step-05` match set, and the Template B PR field were
   all added after the execution stories closed — no one has exercised
   them against a real ClickUp invocation. The case against making it a
   hard go-condition: a second run requires a willing pilot team, a
   workspace, and a sprint window; blocking GO on it adds calendar risk
   and delays broader adoption. Position: declare CONDITIONAL GO and
   treat the first non-bmad-mcp-server adoption team as the de facto
   second pilot, capturing the risk in action item A3. If that team
   encounters a blocking defect in the post-5-7 state, the EPIC-5
   verdict is revisited.

4. **EPIC-4 (non-ClickUp agent audit) is the highest-urgency pre-rollout
   work.** EPIC-5 §Outcomes do not depend on EPIC-4, but the EPIC-3
   retro action item A5 and the EPIC-5 §Dependencies note ("EPIC-4
   strongly recommended to prevent surprise file writes during pilot")
   are both unresolved. A second team running the flow without EPIC-4
   could encounter PM / Architect / UX agents writing sprint or story
   files into their repo under unexpected conditions. Assessment: EPIC-4
   is a **go-condition-soft** — not blocking the GO verdict (the pilot
   proved the ClickUp flow works end-to-end), but strongly recommended
   before broad team adoption. EPIC-4 is captured as action item A1 with
   an explicit deadline tied to the second team's adoption.

## Go/no-go decision

**CONDITIONAL GO — proceed once the conditions below are captured as
active action items.**

The pilot satisfied all three EPIC-5 exit criteria and all six EPIC-5
outcomes. One story was completed end-to-end in ClickUp with a linked PR
and no story / sprint files created in the pilot repo. The 22-entry
friction log was captured and fully translated: 16 entries into concrete
skill / config / planning-artifact refinements, 2 into team-facing
documentation, 2 into human-only operator-awareness items, and 2
intentionally left out-of-scope. The team-facing quickstart
(`docs/clickup-quickstart.md`) exists and covers all required sections.
The flow is proven to produce the contracted ClickUp artifacts (M1 / M2
comments, status transition, PR link) on a real workspace and a real
pilot repo.

Residual risks are real but bounded. The cwd deviation workaround was
required in both execution stories; the post-5-7 `.bmad-pilot-marker`
cwd-assertion guard provides a runtime halt if the deviation recurs
silently, but the guard has not yet been exercised against a live
invocation. The DS trigger-dispatch gap means Claude Code CLI users must
walk skill steps directly rather than via the `CS` / `DS` TOML trigger
codes — a documented workaround, not a functional blocker. EPIC-4 is not
started; any team adopting the flow before EPIC-4 closes accepts the risk
that non-ClickUp agents may write unexpected files into their repo.

A second pilot run against the post-5-7 skill state is recommended — to
exercise the cwd-assertion guard, the broadened review-status match set,
and the Template B PR field under real conditions — but this is not a hard
go-condition. The first non-bmad-mcp-server adoption team assumes this
verification role.

**Conditions:**

- EPIC-4 kick-off initiated before the first non-bmad-mcp-server team
  adopts the flow (action item A1). Teams may adopt while EPIC-4 is in
  progress, but should not adopt after EPIC-4 has been blocked or
  abandoned without a recorded decision.
- DS trigger-dispatch confirmed or formally documented as
  not-supported in Claude Code CLI before declaring the TOML-dispatch
  path production-ready (action item A4).
- Any team wanting the cross-list layout (stories in sprint list, epic
  in Backlog) must confirm the "Tasks in Multiple Lists" ClickApp toggle
  is ON before running `clickup-create-story` (action item A2).

**What happens next:** Teams can adopt the BMAD+ClickUp flow immediately,
using `docs/clickup-quickstart.md` as the entry point. The first adopting
team's experience constitutes the de facto second pilot run (action item
A3). EPIC-4 stories (4-1 through 4-6) should be created and started
before the second team adopts the flow.
