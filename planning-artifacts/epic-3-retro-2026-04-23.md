## What went well

### 1. Spec discipline produced narrow, auditable diffs

Every story arrived with an exhaustive AC list (12–16 ACs each) that explicitly pinned protected paths via `git diff` invariants. Concretely: each story re-asserted that `BMAD-METHOD/`, `src/tools/clickup/`, specific TypeScript files, and various config files had to stay byte-unchanged. This produced diffs that were mostly readable as "one file, ~50–200 lines, zero unintended touches." When someone did drift out of scope (see challenges below), the reviewer caught it every time because the invariant was already written down.

### 2. Incremental skill construction with frozen predecessors

Stories 3-2 through 3-8 each added one step file (`step-01` through `step-07`) to the skill, with every prior step file explicitly marked byte-unchanged in the current story's ACs. This meant no "skill-wide refactor" mid-epic — every landed step file stayed stable until the whole skill was assembled. Story 3-9's TOML wiring then activated the whole thing atomically.

### 3. Counter-isolation invariant held across 3 stories

Story 3-7 introduced a `{assumption_count}` variable with a grep-based invariant that forbids step N from reading step M's counters. Stories 3-7, 3-8, and subsequent reviews all re-verified the "0 matches" grep. The invariant traveled cleanly through three story boundaries without regression — evidence that the pattern is sustainable.

### 4. The 2.7 foundation paid off at 3-9

Story 2.7 extended `findBmmSkillsRoot` to scan `src/custom-skills/`, making any skill placed there immediately discoverable. By the time story 3-9 landed the TOML wiring, `clickup-dev-implement` was already loadable via `bmad({operation:"read", workflow:"clickup-dev-implement"})` — the entire epic could be verified end-to-end *before* the menu routing activated. Story 3-9 collapsed to a 9-line TOML diff.

### 5. Adversarial code review caught real bugs

The 3-layer code-review workflow (Blind Hunter + Edge Case Hunter + Acceptance Auditor) surfaced both correctness bugs (story 3-2 BS-1: URL query strings not handled) and commit-discipline issues (stories 3-5 R5-2, 3-8 R8-1, R8-2: out-of-scope files bundled into feature commits). The Acceptance Auditor's byte-for-byte comparison against AC-specified file contents is an especially high-signal lens for TOML/markdown-only stories.

## What didn't go well

### 1. Commit-scope drift bundled unrelated edits into feature commits

Two distinct incidents, same root cause. Commit `00e352d` (nominally story 3-5) touched `step-02-task-fetch.md` outside the story's scope — that hunk landed 2 stories early. Commit `2362c42` (nominally story 3-8) bundled a step-04 edit from story 3-5's R5-1 finding plus retroactive planning-artifact edits for stories 3-3 and 3-5 plus a 309-line draft of story 3-9 itself.

**Why:** When fixing a review finding or drafting the next story during an active commit, both changes flowed into the same `git add` cycle. No process step forced a "stage only this story's files" check before `git commit`.

**Impact:** Both incidents required Review Findings entries (R3-1, R5-2, R5-3, R8-1, R8-2) explaining the history. The content was correct; the blast radius was just larger than the story's AC #9 byte-unchanged invariants claimed. Final clean commits (`df05376`, `6555c72`) landed, but the earlier noisy commits remain in history.

**Systemic fix:** See action item **A1** below.

### 2. Review Findings sections were empty for 2 stories

Stories 3-4 (`planning-artifact-reader`) and 3-6 (`status-transition-helper`) were marked `done` in sprint-status but their `### Review Findings` sections still read `(to be filled during code review)`. Either the code review happened off-file and the template was never back-filled, or the review step was skipped for those stories.

**Impact:** Retrospective signal loss — we have no record of what the reviewer examined for those stories. For pilot-ready handoff (EPIC-5), a future operator cannot retrace what was audited. Minor QA risk only (both stories ship prompt content; runtime is exercised by every DS invocation).

**Systemic fix:** See action item **A2** below.

### 3. Inconsistent review-findings format across the epic

Three different formats in use: markdown table with `ID | Category | Finding | Resolution` columns (stories 3-1, 3-2, 3-3, 3-5, 3-8), checkbox-list with `[Review][Defer]` / `[Review][Patch]` tags from `bmad-code-review` output (stories 3-7, 3-9), and empty placeholder (stories 3-4, 3-6). The `bmad-code-review` workflow itself produces the checkbox format; the table format appears to be a pre-workflow convention from earlier stories.

**Impact:** Pattern-matching across stories for recurring findings (e.g., "how often does scope drift appear?") requires format-aware parsing. Not a blocker today; becomes friction at scale.

**Systemic fix:** Adopt the `bmad-code-review` checkbox format consistently. See action item **A3** below.

### 4. EPIC-3 exit criterion "completes a real story end-to-end" was deferred, not delivered

The epic's own exit criteria include "Dev agent completes a real story end-to-end: fetches, implements, comments progress, transitions status." Every story verified its step file in isolation (AC-level verification); none of the 9 stories invoked the fully assembled skill against a real ClickUp task. That verification is now parked in EPIC-5 (`5-5-dev-implements-pilot-story`).

**Why:** Intentional, per the epic's non-destructive framing — end-to-end verification requires a pilot project (EPIC-5), and the epic's stories were structured to make the skill *loadable and correct* before anyone invokes it live. But the exit criterion reads as if the verification happens inside EPIC-3.

**Impact:** A reader of the EPIC-3 file alone can't tell what "done" means. The retrospective is the first place the gap is called out. This is a documentation mismatch, not a scope failure.

**Systemic fix:** Update EPIC-3's exit criteria to say "skill is loadable and AC-verified; end-to-end run is exercised in EPIC-5 pilot." See action item **A4** below.

### 5. Latent platform risks accumulated from story 2.7 into story 3-9

Two cross-story defer items surfaced during 3-9's review:

- `findBmmSkillsRoot` probes `['src/bmm-skills', 'bmm-skills', 'src/custom-skills', 'custom-skills']` in order and stops at the first hit. Any future contributor who adds `src/bmm-skills/` will silently stop finding `src/custom-skills/`, breaking both CS and DS routing with no test failure.
- `BMAD_GIT_AUTO_UPDATE=true` (default) silently pulls upstream `customize.toml` changes. If upstream renames the `DS` code (e.g., `DS → DI`), this repo's `DS` override converts from "replace" semantics into "append new code" semantics, shipping both old and new routing.

Both are pre-existing from story 2.7 and both are now logged in `planning-artifacts/deferred-work.md` under the 3-9 review section. Neither is an EPIC-3 bug, but EPIC-3 inherited and doubled the blast radius of both.

**Systemic fix:** See action item **A5** below.

## Key insights

1. **Non-destructive customization works.** The entire epic shipped against upstream BMAD-METHOD with zero edits to `BMAD-METHOD/`. The combination of `_bmad/custom/*.toml` overrides + `src/custom-skills/` + `findBmmSkillsRoot` discovery is a reusable pattern for future agent customizations (relevant to EPIC-4).

2. **Step-file architecture is worth it.** 7 step files, each under 100 lines, each with a single responsibility, each verifiable in isolation. When the counter-isolation invariant was introduced mid-epic (story 3-7), it was enforceable via a single grep because the step files were small and had well-defined variable surfaces.

3. **TOML-only stories are legitimate implementation stories.** Stories 3-9 (and earlier 2-7) ship no code, no tests, no new files — yet they're the stories that *activate* weeks of preceding work. The AC framing "exactly this file, these bytes, these invariants" makes them reviewable even without code.

4. **Spec prescriptive-ness trades off against reviewer judgment.** AC #1 of story 3-9 listed the expected final file contents byte-for-byte. The Blind Hunter subagent flagged the em-dash in the description ("non-ASCII em dash... minor smell") — a finding that was immediately dismissed because the em-dash is *mandated* by the spec. Extremely prescriptive ACs collapse the reviewer's discretion; the reviewer's job becomes diff-vs-spec, not code-vs-judgment.

5. **EPIC-3 and EPIC-2 share 80% of their architecture.** Both epics ship a `src/custom-skills/clickup-*` skill with TOML wiring. Differences are in the step files (`clickup-create-story` has 4 steps; `clickup-dev-implement` has 7). This suggests a reusable template for future `clickup-*` skills (e.g., if EPIC-5's pilot reveals a need for `clickup-triage-comments`).
