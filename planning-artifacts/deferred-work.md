## Deferred from: code review of 3-7-non-blocking-assumption-pattern (2026-04-23)

- `{assumption_count}` is stored as a quoted string (`'0'`, `''`) across frontmatter, rule (c), and instruction 1, but rule (f) and instruction 5 describe "increment by 1" — numeric arithmetic. No coercion rule stated. Systemic with step 4's `{comment_count}`; resolve once the skill formalises a variable-type contract for all step frontmatter keys.
- `addComment` tool-response parsing: the wrapper at `src/tools/clickup/src/tools/task-write-tools.ts` emits `comment_id: ${commentData.id || "N/A"}` inside a text field, not as a structured JSON key. If `commentData.id` is undefined upstream, step 6's instruction 5 will set `{last_assumption_comment_id}` to the literal string `"N/A"` while still incrementing the count. Pre-existing across steps 4/6; fix by either returning a structured response from the wrapper or having all consuming steps validate the extracted value against an ID regex.
- Failure detection protocol in `addComment` consumers is underspecified: the wrapper swallows HTTP errors and returns a success-shaped response whose text starts with `"Error adding comment:"`. Agents must string-match to detect failure, risking false positives (task names containing "Error") or misses. Systemic across steps 4/5/6; resolve alongside the response-parsing fix above.
- Pre-3.8 fallback clause in step 6's instruction 2 introduces a HALT path that contradicts rule (a)'s non-blocking contract for the 3-7 → 3-8 merge window. Transitional by design; remove the clause as part of story 3.8's landing checklist, leaving only the "invoke `step-07-dev-clarification.md` instead" branch.
- Rule (f) assumes all assumption posts precede the M2 summary, but "WHEN TO POST" permits any point between step 3 and step 5 — including post-M2 / pre-step-5. Such late-stage assumptions will post to ClickUp but will not appear in any summary. Either tighten "WHEN TO POST" to pre-M2, or add a "late-assumption trailer" to step 5's status-transition comment.
- Frontmatter default `{assumption_count}: ''` collides with the workflow.md-documented state "write mode was active but no assumption was successfully posted." A reviewer cannot distinguish "step 6 never invoked" from "step 6 invoked and all posts failed." No current downstream step reads `{assumption_count}`, so this is latent; resolve when the M2 summary starts consuming the counter by introducing a dedicated sentinel (e.g., `'-'` for never-invoked, `''` for invoked-no-success).
