# Story 6.1: Add `smol-toml` to runtime dependencies (deps-only PR, lands first)

Status: review

Epic: [EPIC-6: Configurable doc-path resolution (cascade)](../epics/EPIC-6-configurable-doc-path-resolution.md)

> First story in EPIC-6. Pure dependency addition — declares the TOML parser that stories 6.2 (`src/utils/toml-loader.ts`) and 6.3 (`src/utils/doc-path-resolver.ts`) will import, then 6.4 (`resolve-doc-paths` MCP operation) and 6.5–6.7 (skill migrations) consume. No TypeScript, no source files, no docs land here. The package.json + package-lock.json deltas are the entire deliverable.
>
> **Resolves the EPIC-6 open question on TOML library choice in favor of `smol-toml`** — small (~5 kB), ESM-native (matches `"type": "module"` in this repo), actively maintained, MIT-licensed. The alternative `@iarna/toml` is CJS-only and unmaintained since 2022. Pinning the choice now means stories 6.2–6.7 can import `smol-toml` directly without a downstream library swap.
>
> Lands ahead of every other EPIC-6 story so each follow-up can assume the dep exists. The dependency-audit test (`tests/unit/dependency-audit.test.ts`) walks `src/` and rejects any import not declared in `package.json`; without 6.1 first, story 6.2 would fail that audit on its own merits.

## Story

As the **bmad-mcp-server platform maintainer**,
I want `smol-toml` declared as a runtime dependency in `package.json` (with `package-lock.json` regenerated to match) and no other source changes,
so that subsequent EPIC-6 stories can import a TOML parser to read `.bmadmcp/config.toml` and BMAD `_bmad/config.toml` files in a per-key cascade — matching EPIC-6 §Outcomes ("Thin TOML loader `src/utils/toml-loader.ts` wrapping `smol-toml` (new dep)") — without forcing the dependency choice to ride alongside a behavioural change that would entangle review of the lib pick with review of the resolver semantics.

## Dev Notes

### Why `smol-toml` and not `@iarna/toml`

EPIC-6 §Open questions explicitly flagged this as a decision to resolve before coding. The factors:

| Factor               | `smol-toml`                           | `@iarna/toml`                               |
| -------------------- | ------------------------------------- | ------------------------------------------- |
| Module system        | ESM-native (`"type": "module"`)       | CJS only — would force `createRequire` shim |
| Last release         | Active (multiple in 2024–2025)        | Stale since 2022                            |
| Bundle size          | ~5 kB                                 | ~30 kB                                      |
| TOML 1.0 conformance | Yes                                   | Yes                                         |
| License              | BSD-3-Clause / MIT (verify in Task 2) | ISC                                         |
| Transitive deps      | Zero (verify in Task 2)               | Zero                                        |

This repo's `package.json` declares `"type": "module"` and every existing import in `src/` uses ES module syntax. A CJS-only TOML lib would require boilerplate to load (`import { createRequire } from 'node:module'; const require = createRequire(import.meta.url); const toml = require('@iarna/toml');`) and that's exactly the kind of friction story 6.2's "thin loader" should not be carrying.

### Why deps-only, not bundled with story 6.2

EPIC-6's bullet list deliberately splits "Add `smol-toml` to dependencies" from "Implement `src/utils/toml-loader.ts`" so the dependency-add PR is reviewable on its own merits (license, version, supply-chain footprint) without entangling the review with behavioural code. This mirrors the pattern from EPIC-1 story 1.1 (vendoring the ClickUp source as its own commit, before any wiring). It also means a dep audit revert is a one-line revert — no behavioural rollback.

### Why `dependencies`, not `devDependencies`

The `bmad-mcp-server` npm package ships compiled JavaScript from `build/`, and `build/utils/toml-loader.js` (story 6.2) will `import` `smol-toml` at runtime when an end-user installs the published package. `devDependencies` are stripped from the install tree by `npm install --omit=dev` (and by tools like `npm install <package>` in downstream projects). Putting `smol-toml` in `devDependencies` would crash the published binary the first time a user invokes a doc-path cascade.

Compare to `js-yaml` (already a runtime dep) vs `@types/js-yaml` (devDep) — same shape: parser at runtime, types at build time. `smol-toml` ships its own types (verify after install: `node_modules/smol-toml/dist/index.d.ts` or similar), so no `@types/smol-toml` is needed.

### Interaction with the existing prose-driven TOML reads

`src/custom-skills/clickup-create-story/steps/step-02-epic-picker.md` and a few sibling steps already reference `.bmadmcp/config.toml` for pinned-config short-circuits (`pinned_space_id`, `pinned_backlog_list_id`, etc.). Those reads are LLM-prose-driven — the agent reads the file with its `Read` tool and extracts values by reading the TOML as plain text. They do not import any TOML library and are not affected by this story.

EPIC-6 introduces a separate, code-driven cascade for **doc paths** (`prd_path`, `architecture_path`, `epics_path`). The resolver lives in TypeScript (story 6.3) and uses `smol-toml` to parse. Eventually those two paths may converge — a single `bmad` operation that reads any `[table].key` from any cascade layer — but that convergence is explicitly out of scope for EPIC-6 and not preempted by this story.

### Why no new test now

`tests/unit/dependency-audit.test.ts` already enforces "every import in `src/` is in `package.json`". Adding `smol-toml` to `package.json` is sufficient for the audit; no test needs to assert the dep exists. Story 6.2 will land the first `import { parse } from 'smol-toml'` and the first `tests/unit/utils/toml-loader.test.ts`, at which point the audit's coverage closes the loop.

If we wanted a defensive "TOML library consistency" test mirroring the `js-yaml` consistency check (rejects `import .* from 'yaml'`), the natural shape would be "rejects `import .* from '@iarna/toml' | 'toml' | '@ltd/j-toml'`". That guard belongs in story 6.2 alongside the first legitimate `smol-toml` import — landing it here would gate against an absence with no offsetting positive assertion.

### Lockfile churn expectations

`smol-toml` is a leaf package — no transitive runtime deps as of 1.4.x. The expected `package-lock.json` delta is:

1. One new `node_modules/smol-toml` entry under `packages` with `version`, `resolved` (npm registry tarball URL), `integrity` (sha512), `license`.
2. Reference to `smol-toml` in the root project's `dependencies` block of the lockfile.

Anything more — say, a transitive that pulled in another TOML lib or a polyfill — should halt the work and surface to the platform maintainer.

### File-system layout impact

`node_modules/smol-toml/` lands locally after `npm install`. `node_modules/` is gitignored, so this is invisible in the diff. CI re-runs `npm install` per the `ci.yml` workflow, so no docker / pipeline changes are needed.
